# 定时任务配置说明

## 当前配置

| 任务名称 | 当前频率 | 配置位置 | 建议频率 |
|---------|---------|----------|----------|
| 竞拍延期检查 | 5秒 | AuctionScheduler.java:53 | 5-15秒 |
| 竞拍结算 | 10秒 | AuctionScheduler.java:34 | 10-30秒 |
| 竞拍自动开始 | 30秒 | AuctionScheduler.java:70 | 30-60秒 |
| 数据一致性检查 | 5分钟 | AuctionScheduler.java:85 | 5-15分钟 |

## 如何调整频率

### 1. 竞拍延期检查（当前5秒）
```java
// 文件：AuctionScheduler.java 第53行
@Scheduled(fixedDelay = 5000, initialDelay = 3000)  // 5000毫秒 = 5秒
```
**建议调整：**
- 开发环境：10-15秒
- 生产环境：5秒（保证实时性）

### 2. 竞拍结算（当前10秒）
```java
// 文件：AuctionScheduler.java 第34行
@Scheduled(fixedDelay = 10000, initialDelay = 5000)  // 10000毫秒 = 10秒
```
**建议调整：**
- 开发环境：30秒
- 生产环境：10秒

## 如何临时禁用某个任务

在对应的任务方法上注释掉 `@Scheduled` 注解：
```java
// @Scheduled(fixedDelay = 5000, initialDelay = 3000)  // 注释掉即可禁用
public void checkAuctionDelay() {
    // ...
}
```

## 性能影响分析

### CPU 使用
- 当前4个定时任务，平均CPU占用 < 1%
- 主要消耗在数据库查询上

### 数据库负载
- 每分钟约12次查询（延期5秒x12 + 结算10秒x6）
- 都是索引查询，性能影响很小

### 内存占用
- 定时任务本身内存占用可忽略
- 主要是业务数据处理

## 业务重要性

### 🔴 必须开启的任务
1. **竞拍延期检查**：保证拍卖公平性
2. **竞拍结算**：生成订单，完成交易

### 🟡 可选任务
3. **竞拍自动开始**：可手动触发
4. **数据一致性检查**：开发阶段可关闭

## 监控建议

在生产环境中，建议添加监控：
```java
if (count > 0) {
    log.info("竞拍结算定时任务执行完成，结算数量: {}", count);
    // 发送到监控系统
    metricsService.record("auction.settlement", count);
}
```

## 注意事项

⚠️ **不要完全禁用所有定时任务！**
- 竞拍将无法自动结束
- 订单无法自动生成
- 用户将无法完成交易流程

✅ **建议的开发配置：**
- 竞拍延期：15秒
- 竞拍结算：30秒
- 其他：保持默认或禁用
